### Biggest blocker: Circular dependency `msgpack_rpc ↔ os`

`bazel build //src/nvim/...` fails at analysis with:
```
cycle in dependency graph:
.-> //src/nvim/msgpack_rpc:msgpack_rpc
|   //src/nvim/os:os
`-- //src/nvim/msgpack_rpc:msgpack_rpc
```

The concrete cause: `src/nvim/os/input.c` includes `nvim/msgpack_rpc/channel.h` and uses `ch_before_blocking_events` (defined there). Simultaneously, `msgpack_rpc/` code depends on `os/` APIs. This is a genuine architectural cycle—`msgpack_rpc` and `os` are mutually dependent.

Gazelle_cc correctly identifies both directions as `implementation_deps`, but Bazel's DAG requirement prohibits even `implementation_deps` cycles. Resolving this requires either: (a) splitting `ch_before_blocking_events` into a separate no-dep header, (b) merging `os` and `msgpack_rpc` into a single package, or (c) moving `os/input.c` into a package that can depend on both—all of which exceed the "minor edits only" constraint.
